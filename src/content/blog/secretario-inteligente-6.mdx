En el artículo anterior de esta serie te conté algo que se volvió evidente mientras construía mi sistema de agentes. El problema no era hacer un agente más potente. El problema era **quién controla lo que hace el agente**. Cuando empiezas a conectar agentes con herramientas reales —servidores, automatizaciones, APIs o repositorios— aparece un riesgo muy claro: **un único agente con acceso total puede hacer demasiado**. Y ese fue el momento en el que el proyecto cambió de dirección. En lugar de intentar construir un agente omnipotente, empecé a pensar en algo distinto: una **arquitectura de gobernanza de agentes**.
## El problema de los agentes con herramientas
Muchos sistemas de agentes que ves hoy siguen un patrón bastante simple. Un usuario hace una pregunta, el *LLM* interpreta la situación, llama a una herramienta (*tool*) y ejecuta una acción. Si lo dibujamos de forma sencilla sería algo así:
```

Usuario
↓
LLM
↓
Tools
↓
Acción

```
En una demo esto es espectacular. Pero cuando conectas ese sistema a infraestructura real —servidores, despliegues, automatizaciones— empiezan los problemas. Un único sistema interpreta, decide y ejecuta. Todo al mismo tiempo. **Demasiada responsabilidad concentrada en un solo punto.**
> Imagina que en tu empresa una sola persona pudiera analizar la situación financiera, decidir una inversión y ejecutar una transferencia bancaria sin ningún control. Probablemente no te sentirías muy cómodo con ese sistema.
Con los agentes ocurre exactamente lo mismo.


## De un agente a un sistema de agentes
La solución que empecé a explorar fue separar responsabilidades. En lugar de un único agente con todo el poder, el sistema se divide en varias etapas. Algo parecido a esto:
```

Human
↓
Analyzer
↓
Planner
↓
Human approval
↓
Executor
↓
Auditor

````
Cada agente tiene un rol muy claro.
### Analyzer
El **Analyzer** interpreta lo que está ocurriendo. Por ejemplo:
- estado de los servidores
- contenedores Docker
- errores de red
- alertas del sistema
Su trabajo es **entender la situación**. No ejecutar nada.
### Planner
El **Planner** propone posibles acciones. Por ejemplo:
- reiniciar un contenedor
- limpiar logs
- revisar un dominio
- ignorar un falso positivo
El planner **sugiere planes**, pero no ejecuta.
### Human approval
Antes de ejecutar acciones potencialmente peligrosas, el sistema pide confirmación. Esto mantiene al humano dentro del circuito de control.
### Executor
El **Executor** es el único agente que puede ejecutar acciones reales. Por ejemplo:
```bash
docker restart container
docker compose up
systemctl restart
````

Pero hay una regla muy importante. **El executor no piensa.** Solo ejecuta acciones autorizadas.

### Auditor

El **Auditor** verifica lo que ocurrió después de cada acción. Por ejemplo:

* si el problema se resolvió
* si aparecieron efectos secundarios
* si la acción fue realmente necesaria
  Esto introduce algo clave en sistemas reales: **trazabilidad**.


## La capa de gobernanza de agentes
A medida que el sistema fue creciendo, empezó a quedar claro que no estábamos simplemente añadiendo herramientas a un agente. Lo que estaba emergiendo era algo distinto: una **capa de gobernanza**. Una capa que controla:

* quién puede actuar
* cuándo puede actuar
* qué herramientas puede usar
* cómo se registran las acciones
  En lugar de permitir que un único agente piense y actúe directamente sobre el sistema, la arquitectura separa responsabilidades entre distintos agentes especializados: análisis, planificación, aprobación, ejecución y auditoría. Esto introduce algo fundamental cuando los agentes interactúan con sistemas reales: **control y trazabilidad**. La siguiente figura resume esta arquitectura.

## Tres capas del sistema
Si te fijas en el diagrama, la arquitectura se divide en tres capas principales.
| Capa | Función |
|-----|------|
| **Agentes de negocio** | OpenCTO, OpenCIO, OpenCMO |
| **Sistema de gobernanza** | Analyzer → Planner → Human → Executor → Auditor |
| **Infraestructura** | Memory, Tools, Skills |
### Agentes de negocio
Los agentes superiores representan **roles reales dentro de una organización**. Por ejemplo:
- **OpenCTO** → infraestructura
- **OpenCIO** → conocimiento
- **OpenCMO** → marketing
Cada uno de ellos puede utilizar el sistema de agentes, pero **bajo las mismas reglas de gobernanza**.
### Sistema de gobernanza
Aquí ocurre el flujo de decisión real del sistema.
```
Analyzer → Planner → Human approval → Executor → Auditor
```
Este patrón introduce algo muy importante: **control del poder del agente**.
### Infraestructura
En la capa inferior aparecen los recursos que los agentes pueden utilizar:
- **Tools** → APIs o acciones ejecutables
- **Skills** → capacidades reutilizables
- **Agent Memory** → memoria contextual
- **Memory Vault** → almacenamiento persistente
Las tools y las skills son las capacidades reales del sistema, pero **no se ejecutan directamente**. Siempre pasan por la capa de gobernanza.


## Un pequeño apunte técnico para los curiosos
Si quisieras modelar este tipo de arquitectura desde un punto de vista matemático, podrías pensar en cada agente como una función dentro de un flujo de decisión. Simplificando mucho:
$$
Action = Executor(Approve(Plan(Analyze(Event))))
$$
Donde:
- $Event$ representa una señal del sistema (por ejemplo un error en un servidor)
- $Analyze()$ interpreta la situación
- $Plan()$ genera posibles acciones
- $Approve()$ valida la acción
- $Executor()$ ejecuta la operación
> Piensa en esto como una cadena de ayudantes. El primero entiende el problema, el segundo propone una solución, el tercero decide si esa solución es segura y el último ejecuta la acción.
Este pequeño detalle cambia completamente la arquitectura.


## Por qué esta arquitectura importa
La mayoría de sistemas de agentes actuales siguen este patrón:
```
think → execute
```
Es rápido. Pero también es arriesgado. La arquitectura que estamos construyendo intenta sustituir ese modelo por otro:
```
think
 ↓
plan
 ↓
approve
 ↓
execute
 ↓
audit
```
Este patrón aparece en muchos sistemas críticos como:
- aviación
- medicina
- sistemas industriales
Nunca se permite que una sola entidad piense y actúe sin control.
## En qué punto estamos
Ahora mismo el sistema ya tiene varias piezas funcionando:
- monitorización del servidor
- alertas enviadas a Slack
- agentes estructurados en OpenClaw
- separación inicial de roles
La capa de gobernanza empieza a conectar todo eso. Todavía no es el sistema final, pero ya empieza a mostrar algo interesante. El sistema deja de ser un conjunto de scripts y empieza a parecerse a **una arquitectura de control para agentes**.
